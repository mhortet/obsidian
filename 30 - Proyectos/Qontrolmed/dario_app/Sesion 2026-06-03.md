## Foco

- endurecimiento de acceso;
- base minima de roles;
- auditoria de eventos sensibles.

## Cambios cerrados

- login endurecido: se bloquea el acceso implicito sin contrasena salvo
  bootstrap explicito por `QONTROLMED_ALLOW_PASSWORDLESS_LOGIN=1`;
- modelo operativo actual fijado en dos roles:
  - `Administrador`;
  - `Tecnico`;
- compatibilidad actual sin migracion de roles:
  - `es_tecnico = 1` => `Tecnico`;
  - `es_tecnico = 0` => `Administrador`;
- mantenimiento de usuarios preparado para usar `rol` cuando exista columna en
  BD;
- tabla y servicio base de auditoria preparados;
- auditoria validada para:
  - login correcto;
  - login erroneo;
  - cierre de informe Preventivos;
  - PDF de Preventivos;
  - cierre de informe Quirofanos;
  - PDF de Quirofanos;
  - libro de quirofano.

## Scripts y migraciones relevantes

- `bd/migracion_gestion_auditoria.sql`: crea la tabla `auditoria_eventos`;
- `bd/gestion_auditoria_eventos.sql`: definicion base de esa tabla;
- `bd/migracion_gestion_roles.sql`: deja el rol explicito en BD cuando se
  quiera activar ese paso;
- `bd/gestion_usuarios.sql`: esquema base actualizado con columna `rol`.

## Validaciones realizadas

- login correcto entra;
- login incorrecto avisa correctamente;
- un usuario con `es_tecnico = 0` desaparece de selectores de tecnico en
  Preventivos y Quirofanos;
- la tabla `auditoria_eventos` recoge:
  - acceso correcto;
  - acceso erroneo;
  - cierre e impresion de informe Preventivos;
  - cierre e impresion de informe Quirofanos;
  - impresion de libro de quirofano y sus informes.

## Riesgos o notas

- hay cambios locales funcionales y documentales acumulados sobre `main`;
- existe un cambio local en `pdf_assets/qontrolmed/logo.png` usado para trabajo
  de campo real y no debe tocarse;
- la migracion de roles explicitos puede esperar; hoy no es necesaria para la
  semana de prueba real;
- la auditoria ya funciona, pero la tabla debe existir en la BD para persistir
  eventos.
